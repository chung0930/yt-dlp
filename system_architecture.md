# 系統架構與通訊機制設計

## 1. 總覽

本專案旨在建立一個基於 Termux 後端與 GitHub Pages 前端的分散式 YouTube/YouTube Music 下載系統。使用者透過前端介面提交下載請求，後端負責解析、下載、處理檔案，並將結果回傳至前端供使用者下載。整個系統將以「一鍵部署」為目標，簡化設定流程。

## 2. 技術選型

| 元件 | 技術/工具 | 說明 |
|---|---|---|
| **後端 (Termux)** | Python 3.x | 主要程式語言 |
| | `Flask` (輕量級 Web 框架) | 處理 HTTP 請求與回應，提供 RESTful API |
| | `yt-dlp` | 核心下載工具，支援多種影音平台 |
| | `ffmpeg` | 處理影音轉碼、合併音軌與畫面、封面嵌入 |
| | `pigz` | 高效能平行壓縮工具，用於打包多個檔案 |
| | `serveo.net` (或類似服務) | 提供內網穿透，使 Termux 後端可從外部網路存取 |
| **前端 (GitHub Pages)** | HTML5, CSS3, JavaScript | 靜態網頁，提供使用者介面 |
| | `Fetch API` / `Axios` | 進行非同步 HTTP 請求與後端通訊 |
| | `Bootstrap` / `Tailwind CSS` | 提升 UI 美觀與響應式設計 |
| **通訊協定** | HTTP/HTTPS | 前後端之間的主要通訊方式 |
| | WebSockets (可選，用於進度即時更新) | 考慮用於即時進度回報，但初期可能採用輪詢 (polling) 簡化 |
| **部署** | GitHub Pages | 託管前端靜態檔案 |
| | Termux | 運行後端伺服器 |

## 4. 通訊機制與流程

### 4.1. 後端啟動與網路穿透

1.  使用者在 Termux 中執行 `server.py`。
2.  `server.py` 會檢查並自動安裝 `yt-dlp`, `ffmpeg`, `pigz` 等依賴。
3.  `server.py` 啟動一個 Flask 應用作為 HTTP 伺服器，監聽特定埠 (例如 5000)。
4.  `server.py` 嘗試透過 `ssh -R 80:localhost:5000 serveo.net` 命令建立一個 SSH 隧道，將本地 5000 埠映射到 `serveo.net` 提供的隨機子域名 (例如 `random-string.serveo.net`) 的 80 埠。
5.  後端將這個公開的 URL 顯示給使用者，使用者需將此 URL 配置到前端的 JavaScript 檔案中。

### 4.2. 前端登入與請求

1.  使用者開啟 GitHub Pages 上的前端網頁。
2.  前端載入 `users.js` 檔案，其中包含預設的用戶名和密碼。
3.  使用者在登入介面輸入憑證，前端進行客戶端驗證。
4.  登入成功後，使用者可以選擇下載模式並輸入 YouTube/YouTube Music 連結。
5.  前端將下載請求 (包含連結、下載模式、用戶 token) 以 POST 請求發送到後端提供的公開 URL 的 `/download` 端點。

### 4.3. 後端處理下載請求

1.  後端 `/download` 端點接收到請求。
2.  後端驗證請求的合法性 (例如，檢查用戶 token，儘管前端驗證已足夠，後端可增加一層簡單的保護)。
3.  根據請求中的下載模式，後端調用 `yt-dlp` 和 `ffmpeg` 執行下載和處理任務。
    *   **音樂專輯下載**：使用 `yt-dlp` 下載音訊，嵌入封面 (1:1 裁剪)，添加音樂名稱、作曲人、專輯、發行日期等 metadata。檔案儲存路徑為 `/sdcard/Download/專輯名稱/01 - 歌曲名稱.mp3`。
    *   **單音樂下載**：同上，但為單一檔案。
    *   **播放清單音樂下載**：同上，但將整個播放清單視為一張專輯，不分資料夾，直接儲存到 `/sdcard/Download/播放清單名稱/01 - 歌曲名稱.mp3`。
    *   **1080p 影片下載**：使用 `yt-dlp` 下載最佳 1080p 影片流和最佳音訊流，然後使用 `ffmpeg` 將兩者合併為 MP4 檔案。
    *   **影片播放清單下載**：同上，但為多個影片。
4.  下載過程中，後端會將進度資訊寫入一個臨時檔案或記憶體中，以便前端查詢。
5.  下載完成後，後端根據檔案數量決定處理方式：
    *   **單一檔案 (單音樂/單影片)**：後端將檔案路徑儲存，並回傳一個包含下載連結的 JSON 回應給前端。
    *   **多個檔案 (專輯/播放清單)**：後端使用 `pigz` 將所有下載的檔案壓縮成一個 `.tar.gz` 檔案，然後回傳一個包含該壓縮檔下載連結的 JSON 回應給前端。

### 4.4. 前端顯示進度與下載

1.  前端在提交下載請求後，會定期向後端的 `/status` 端點發送請求，查詢特定任務的下載進度。
2.  後端 `/status` 端點讀取進度資訊並回傳給前端。
3.  前端接收到進度資訊後，更新 UI 上的進度條或文字。
4.  當後端回傳下載連結時，前端顯示一個下載按鈕。使用者點擊按鈕即可從後端直接下載檔案 (單一檔案或壓縮包)。

## 5. 檔案儲存與部署

*   **後端程式碼**: `server.py` 將包含所有後端邏輯，部署在 Termux 的 `/home/ubuntu/` 目錄下。
*   **下載檔案**: 所有下載的影音檔案將儲存到 Termux 上的 `/sdcard/Download` 目錄。
*   **前端程式碼**: `index.html`, `style.css`, `script.js`, `users.js` 將部署在 GitHub Pages 倉庫的根目錄下。
*   **一鍵部署**: `server.py` 將包含自動安裝依賴、啟動 Flask 伺服器和建立 `serveo.net` 隧道的邏輯，實現「一個檔案，部署全部」的目標。

## 6. 安全考量

*   **前端登入**: 由於 GitHub Pages 是靜態頁面，登入驗證將完全在客戶端進行。`users.js` 檔案將公開，因此不應包含任何敏感資訊。這僅用於提供一個基本的用戶管理介面，而非嚴格的安全驗證。
*   **後端存取**: `serveo.net` 提供的 URL 是公開的，任何知道 URL 的人都可以存取。雖然可以透過簡單的 token 驗證來增加一層保護，但由於是個人使用，且檔案儲存在手機本地，風險相對可控。建議使用者在不使用時關閉 Termux 伺服器。

## 7. 進度回報機制

初期將採用前端輪詢 (polling) 後端 `/status` 端點的方式來獲取下載進度。後端會將 `yt-dlp` 的輸出解析，提取進度資訊並儲存，供前端查詢。未來可考慮升級為 WebSocket 以實現更即時的通訊。
